Лабораторная работа №1

Разработка сервиса управления личными финансами · FastAPI · PostgreSQL · JWT

Цель работы

Спроектировать и реализовать backend-сервис для учёта доходов и расходов, категорий, бюджетов и целей, с аутентификацией пользователей и REST API для типовых операций CRUD.

Постановка задачи

  • пользователь вводит доходы и расходы, задаёт бюджеты по категориям, ведёт цели;
  • данные хранятся в реляционной БД с явными связями one-to-many и many-to-many;
  • ассоциативная сущность для M2M содержит поле, характеризующее связь (не только внешние ключи);
  • API: регистрация, вход (JWT), профиль, список пользователей, смена пароля;
  • CRUD по сущностям предметной области и выдача вложенных структур там, где нужны связи.

Технологический стек

Backend: Python 3, FastAPI, Uvicorn
БД: PostgreSQL, SQLAlchemy 2.x, Alembic
Безопасность: JWT (python-jose), хэширование паролей (passlib / argon2)
Инфраструктура: Docker Compose (при необходимости), Redis для TTL-токенов (Telegram-flow)
Документация API: OpenAPI / Scalar

Модель данных

Реализовано не менее семи таблиц. Основные сущности:

  • users — пользователь (email, ФИО, хэш пароля, активность, метки времени);
  • finance_accounts — счета пользователя (one-to-many от пользователя);
  • finance_categories — категории дохода/расхода;
  • finance_transactions — операции (связь со счётом и категорией);
  • finance_budgets — лимиты по категориям и периоду;
  • finance_goals — финансовые цели;
  • finance_tags и m2m_transaction_tags — теги и связь many-to-many с транзакциями; в ассоциативной таблице задано дополнительное поле (например, вес релевантности тега к операции).

Связи: пользователь → счета, категории, транзакции, бюджеты, цели, теги (one-to-many); транзакция ↔ тег (many-to-many через промежуточную таблицу с атрибутом связи).

Архитектура приложения

Слои проекта (каталог lab1/gateway/):

  • api/apps/ — маршруты FastAPI по типам запросов (get/post/put/delete), response_model и описания для OpenAPI;
  • core/ — бизнес-логика (auth, users, finance), возврат типизированных DTO из templates/base_models;
  • models/ — ORM-модели SQLAlchemy;
  • templates/ — Pydantic-схемы запросов/ответов и enums;
  • migrations/ — ручная миграция Alembic на создание схемы;
  • schedulers/data_init.py — начальное наполнение БД (демо-пользователь, категории, счёт и т.д.);
  • middlewares/permission.py — проверка Bearer JWT для защищённых маршрутов.

API (кратко)

  • Auth: POST /auth/register, POST /auth/login (OAuth2 password form), смена пароля, при необходимости — Telegram TTL-токены;
  • Пользователи: GET /auth/me, GET /auth/users, GET /users/..., детальный ответ с вложенными счетами, категориями, бюджетами, целями и транзакциями с тегами;
  • Финансы: CRUD по счетам, категориям, транзакциям, бюджетам, целям, тегам; привязка тега к транзакции.

Результаты

  • схема БД соответствует требованиям по числу таблиц и типам связей;
  • реализованы аутентификация JWT и типовые эндпоинты для пользователя;
  • данные и ответы API согласованы с Pydantic-моделями в templates;
  • миграции и сиды обеспечивают воспроизводимый подъём окружения.

Запуск (локально)

cd lab1/gateway
pip install -r requirements.txt
# настроить .env по example.env
alembic upgrade head
uvicorn main:main_app --host 0.0.0.0 --port 8000

При использовании Docker — см. lab1/docker-compose.yml в репозитории.
